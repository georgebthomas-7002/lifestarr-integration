## 🔴 Blockers — must resolve before flipping the switch

### 1. Verify the Mighty webhook is actually delivering events
**Status:** Webhook configured in Mighty (per screenshot 2026-05-07). Two concerns to verify before trusting it:

- ⚠️ **URL has a double slash:** `https://lifestarr-integration.vercel.app//api/webhook` (between `.app` and `/api`). HTTP normalizes most of the time, but worth cleaning to a single slash to be safe.
- ⚠️ **Bearer prefix may be double-prepended:** Mighty's "API Key" field hint says it will *include* the value in the `Authorization` header. If George typed `Bearer 848de3a...` AND Mighty auto-prepends `Bearer ` itself, the actual header sent will be `Authorization: Bearer Bearer 848de3a...` — our `verifyMightyWebhook` in `lib/verify-webhook.ts` will 401-reject this silently. If Mighty just uses the value verbatim, it works.

**How to check both:**

1. Trigger any low-stakes Mighty event (test purchase, post a comment, etc.).
2. Within ~10 seconds, query the DB for a NEW non-test row:
   ```bash
   node -e "require('dotenv').config({path:'.env.local'});
   const {neon}=require('@neondatabase/serverless');
   const sql=neon(process.env.DATABASE_URL);
   sql\`SELECT event_id, event_type, status, error_message, received_at FROM webhook_events WHERE event_id NOT LIKE 'test-%' ORDER BY received_at DESC LIMIT 5\`.then(r => console.log(r))"
   ```
3. If row appears with `status = success` (or `needs_review`): integration is live. ✅
4. If no row: Mighty either hasn't fired or is being 401-rejected. Check Vercel logs:
   ```bash
   vercel logs --prod | grep -i "webhook\|401"
   ```
5. If 401s are showing: edit the Mighty config — try just `848de3a9c93e842ddbf0bdd1b37877305cf2464355c1d1ff99e4fa7ded957214` (no `Bearer ` prefix) since Mighty is documented to wrap the value automatically. If THAT 401s, swap back.

**Fix the URL double-slash either way:**
- In Mighty's webhook config screen, edit the URL field to `https://lifestarr-integration.vercel.app/api/webhook` (one slash). Save.

These don't affect go-live and aren't urgent:

1. **No test suite** — verified via manual end-to-end. For a 1–2 person ops tool that's fine; Phase 11 candidate to add Vitest unit tests for handlers + a webhook→HubSpot integration test.
2. **Engagement-score pruning is sync per-event** — works fine for V1. If any member ever exceeds ~1000 events in 30 days, pruning gets slow. Move to a daily cron.
3. **Cascade rules on `needs_review_queue.webhook_event_id`** — currently `set null` on parent delete. Probably correct (preserve audit trail) but document the choice.
4. **Bound `engagement_scores.recent_events` jsonb array size** — pruning keeps it bounded in practice, but a DB-level constraint would be belt-and-suspenders.
5. **Custom domain** — switch from `lifestarr-integration.vercel.app` to e.g. `integrations.lifestarr.com` once Joe's DNS work is done. Cosmetic, builds trust.
6. **Sentry / structured error monitoring** — Vercel Analytics tracks page traffic but not server-side errors granularly. Sentry integration ~10 min if useful.
7. **Backup verification on Neon** — Neon free tier has 7 days of point-in-time recovery built in. Confirm retention matches LifeStarr's recovery requirements.
8. **Cleanup test data** — 13 successful + 2 no-handler test events linger in `webhook_events`. Test contacts in HubSpot (search `lifestarr-test+`). Cosmetic, cleans up the Activity feed.
